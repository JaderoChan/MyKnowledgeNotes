# 知识总结与错误积累-2026-3

## 26-06-09

1. **MacOS** 下创建仅系统托盘菜单程序时可以在 `Info.plist` 文件中设置 `LSUIElement=1`（仅后台/代理应用）。这样当程序隐藏在系统托盘菜单（菜单栏）中时，Dock 区域不会显示应用图标。

    `Info.plist` 示例

    ```xml
    <key>LSUIElement</key>
    <true/>
    ```

2. 静态变量如果在进入程序入口 `main()` 前初始化失败，抛出异常且未被捕获的话会引发 `std::terminal` 并直接终止程序。其中具有 **静态存储期** 的全局对象大多会在 `main()` 之前完成 **零初始化**，而动态初始化可能在 `main()` 之前初始化也可能被推迟。函数内的 `static` 变量（局部静态变量，动态初始化）则是在首次执行到该语句时被初始化。对于动态初始化的静态变量而言，如果首次初始化时抛出异常，则该静态变量被视为 **未完成初始化**，后续再次进入该语句时会尝试重新初始化。
3. **Git** 中更新 **子模块 submodule** 可以通过在指定子模块目录下手动进行拉取

    ```sh
    cd path/to/submodule
    git pull
    ```

    也可以使用 `git submodule update --init --recursive` 初始化并拉取所有子模块，后续也可以通过此命令对子模块进行更新。

    如果需要在克隆存储库期间就一并拉取所有子模块，可以通过 `git clone --recurse-submodules GitUrl...` 实现。

4. **Doxygen** 的 `@details` 指令是用于对 `@brief` 指令进行补充说明的，既它是用于更详细的描述对象功能的说明而不是用于描述对象的实现细节。
5. **C++** 中 `std::move` 可以无条件把对象转成右值（准确说是 **xvalue**），表示：你可以搬走它的资源了。而 `std::forward` 是有条件地“保持原本值类别”转发，主要用于模版编程中实现完美转发。

    **详解**

    - `std::move` 用于触发移动语义，让移动构造/移动赋值有机会被调用。本质是类型转换，不做实际移动；真正移动发生在后续构造或赋值里。

        > **注意点**
        > - 被 move 之后的对象仍“有效”，但值通常不再可预期（只保证可析构、可赋值等）。
        > - 不要对 const 对象期待“移动优化”

    - `std::forward` 用于在模板中保留参数传入时的左右值属性。典型场景是转发引用（也叫万能引用）参数 `T&&`。

6. **C++** 中 `const T&` 可以绑定到左值或常量左值，也可以绑定到右值/临时值（如函数返回值）。当其绑定到临时值时，会将临时值的生命周期延长至这个引用本身的生命周期。 `T&` 只能绑定到非常量左值。`T&&` 通常用于绑定右值/临时值（同样会延长生命周期）或在模板编程中用于实现完美转发（万能引用）。

    **常见坑点**

    ```cpp
    const std::string& bad() {
        return std::string("tmp"); // 返回悬空引用，错误
    }
    ```

    这里临时对象在函数结束就销毁了，不会因为返回引用而继续活着。

7. **C++** 中将临时值赋值给 `const T&` 会比赋值给 `T` 少一次拷贝/移动构造的成本。不过现代 C++ 有拷贝消除（尤其是 **C++ 17** 的保证省略场景），很多时候编译器会把中间临时省掉，所以实际生成的产物中未必存在额外的拷贝/移动。
8. **Qt** 中 **QListWidget** 的 `takeItem()` 方法会返回指定位置元素的指针，并从列表中删除这个元素（但不会释放指针指向的内存，后续可以将方法返回得到的元素再插入回列表）。对于需要完全删除列表元素时（释放内存），推荐先通过 `takeItem()` 或其他 `QListWidget` 的成员方法得到需要被删除的元素指针，然后再进行 `delete`。注意，即使你在外部额外维护了一个列表存放元素指针，也不建议直接对其进行 `delete`，因为通过 `QListWidget` 的成员方法取出元素后再进行删除更加符合其功能语义（将元素的所有权转交给你，再安全删除，避免 `QListWidget` 内部可能存在的悬空引用风险）。
9. **Qt** `drag/drop` 事件中，`accept()` 和 `acceptProposedAction()` 方法的区别在于后者不仅表示接受事件，还明确采用系统当前建议的拖放动作。对于具体行为而言，`acceptProposedAction()` 实际上做了两件事：首先把当前实际动作设为 **proposedAction**（例如 Copy/Move/Link 之一），而后调用 `accept()` 标记事件已被处理。
10. **C++** 中 `dynamic_cast` 用于 **运行时安全地向下转换**。

    **适用场景**

    - 基类指针或引用，实际对象类型不确定时。比如有 Base*，实际可能是 A 或 B，你需要只在它真的是 A 时调用 A 的接口。
    - 需要多态层次里做类型分流时。当你无法通过虚函数设计解决，需要根据真实类型走不同分支时。
    - 需要安全失败机制时

    **特点**

    - 指针版失败会返回 `nullptr`
    - 引用版失败会抛出 `std::bad_cast` 异常

    **不建议的场景**

    - 你已经百分百确定类型。可用 `static_cast`（更快，但不做运行时检查）。
    - 可以用虚函数/接口多态解决。优先多态，少做类型判断，设计更干净。
    - 频繁高性能路径。`dynamic_cast` 有运行时开销，热路径里慎用。

    **前提条件**

    - 基类必须是多态类型（至少有一个虚函数函数，通常虚析构）。
    - 只能在有继承关系的类型间使用。

11. **Qt** 中 **QUrl** 的 `isLocalFile()` 方法对 **本地路径** 返回真，并不区分是文件还是文件夹。

## 26-06-12

1. **C/C++** 中 `getc()` 和 `fgetc` 函数在作用与用法上相同。但 `fgetc()` 保证是一个函数，有固定的函数调用开销；而 `getc()` 可以被实现为宏（也可以是函数），宏展开后可能避免函数调用的开销，但可能会多次求值参数。因此：

    ```c
    // 不要对 getc 传入有副作用的表达式
    getc(fp++);  // 危险：若 getc 是宏，fp++ 可能被求值多次
    fgetc(fp++); // 安全：fgetc 保证是函数
    ```

    推荐使用 `fgetc()`，语义更明确，使用也更安全，并且在现代编译器实现中，其与 `getc()` 几乎没有性能差异。

2. **C/C++** 对一个数组名进行强转后，得到的是强转后的指针地址，而不是指向的内存数据。例如

    ```c
    uint8_t buf[8] = {0};
    uint32_t num = (uint32_t) buf; // 错误
    ```

    `buf` 是数组名，会退化为 `char*` 指针，`(uint32_t)` 强转的是指针的地址值，而不是缓冲区里的数据。正确做法是使用 `memcpy()`：

    ```c
    memcpy(&num, buf, sizeof(uint32_t));
    ```

    直接用指针转换再解引用虽然在很多平台能工作，但属于未定义行为（违反严格别名规则）：

    ```c
    uint32_t num = *(uint32_t*) buf;
    ```

3. **C 标准** 明确规定任何对象都可以通过 `char*` 访问（**C11 §6.3.2.3 / §6.5**），这是严格别名规则的特殊豁免。因此对于 `fread()` 及类似函数可以直接：

    ```c
    uint32_t num = 0;
    FILE* file;
    // ...
    fread((char*) &num, sizeof(uint32_t), file);
    ```

    而不需要引入额外的 `char*` 缓冲区等。

4. **C/C++** `fread()` 函数的二，三参数（`size_t size, size_t nitem`），分别是每个元素的字节大小与要读取的元素个数。返回值是成功读取的元素个数（不是字节数）。
5. **C** 语言也可以使用花括号 `{}` 创建 **块作用域**。可以用于限制临时变量的生命周期，避免命名冲突。此外 `switch` 的 `case` 本身不引入作用域，当需要在 `case` 中创建临时变量时，也要使用 `{}` 创建块作用域。
6. 对于多语句宏，需要使用 `do { ... } while(0)` 写法，其主要是避免在 `if-else` 场景下出错。例如

    ```c
    #define MACRO_A(x) { do_something1(); do_something2(); }             // 不推荐
    #define MACRO_B(x) do { do_something1(); do_something2(); } while(0) // 推荐

    // 编译出错：因为 MACRO_A(X) 后的分号截断了 if 语句导致 else 找不到匹配的 if。
    // 如果不添加分号当然可以编译通过，但是语义不完整，不推荐!
    if (condition)
        MACRO_A(x);
    else
        do_something();

    // 正确：即使在宏后面添加分号也能正常工作。
    if (condition)
        MACRO_B(x);
    else
        do_something();
    ```

7. 在局部作用域可以通过 `#define` `#undef` 来临时声明一个宏，并且不会污染其他区域。
8. **C99** **VLA 数组**

    ```c
    int rows = 3, cols = 4;
    int (*arr)[cols] = malloc(rows * sizeof(*arr));

    // 直接用二维下标
    arr[i][j] = value;
    int v = arr[i][j];

    free(arr);
    ```

    对于上面的 VLA 数组 `arr`，其类型是 `int (*)[cols]`（运行时才确定列数的数组指针）。这种类型不能作为函数参数的类型声明（函数签名在编译期确定）。

9. **C** 语言中，任意指针到 `void*` 与 `void*` 到任意指针的转换都可以是隐式的，无需强转。但在 **C++** 中，后者（`void*` 到任意指针）必须进行强转。也就是说

    ```cpp
    void* p = malloc(100);
    int* q = p;                    // 只有 C 中才可行，C++ 中报错。
    int* q = static_cast<int*>(p); // 正确!
    ```

    此外，使用 **C++ 编译器** 时，即使此类代码使用 `extern "C"` 进行声明了，仍然需要进行强转，具体原因见下一条笔记。

10. **C/C++** 中 `extern "C"` 只影响链接符号的名称修饰（name mangling），让 C++ 编译器以 C 的方式处理符号名，使得 C 和 C++ 代码可以互相链接。

    但类型系统和语法规则仍然是 C++ 的，编译器还是用 C++ 规则解析代码。所以 `void*` 不能隐式转换到其他指针类型这条规则依然成立。
