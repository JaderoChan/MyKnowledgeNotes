# 知识总结与错误积累

## 2026-03-21

- 不要在容器可能扩容的情况下保存其元素指针。如果必须保存指针，要么 `reserve`，要么用 `std::list`/`std::deque`（不会因插入而移动已有元素），要么用 `std::vector<std::unique_ptr<T>>`。

    当容器在可能扩容时保存其元素指针以进行其他操作的情况下，如果容器发生内存重分配扩容，那么原元素指针将成为野指针，此时再对其指向内容进行操作会引发异常行为。

- 不要引用捕获一个局部指针。即使局部指针指向的内容是堆分配的，但脱离局部作用域后，局部的指针变量被销毁，此时捕获的引用将成为悬空引用。在这种情况下，应该按值对指针变量进行捕获。
- `scanf()` 读取失败后，应该通过 `while (getchar() != '\n');` 清空缓冲区。否则无效字符残留在缓冲区中，会影响下一次的 `scanf()`。
- **C++** 中 **POD**（**Plain Old Data**，**平凡类型** 并且 **标准布局类型**）类型成员**不会自动零初始化**。
- **C++** 平凡类型：

    - 有平凡的默认构造/析构函数
    - 有平凡的拷贝/移动构造和赋值操作
    - 不能有虚函数或虚基类
    - 不能有类内初始化

- **C++** 标准布局类型：

    - 所有非静态成员具有相同的访问控制权限
    - 没有虚函数和虚基类
    - 所有非静态成员和基类也是标准布局类型
    - 没有引用类型的非静态成员

1. `CMAKE_CURRENT_LIST_DIR` 和 `CMAKE_CURRENT_SOURCE_DIR` 区别，前者依赖于当前正在执行的 `CMake` 脚本路径，而后者依赖于 `project` 或 `add_subdirectory` 时的 `CMake` 脚本路径。
2. 了解了以下 **CMake 生成器表达式**：`$<TARGET_FILE:target_name>`，`TARGET_FILE_NAME` 与 `TARGET_FILE_DIR`。
3. `cv::Mat` 是引用计数的。
4. 常量引用的形参接受右值实参时会延长右值的生命周期。
5. **CMake** 的 `copy_directory_if_different` 指令是 `CMake 3.26` 加入的。
6. 理解 **单配置生成器** 与 **多配置生成器**（如 **Visual Studio**，**XCode** 生成目标的路径。单配置生成器的生成目标路径通常与 `CMAKE_RUNTIME_OUTPUT_DIRECTORY` 变量一致，而多配置生成器则通常为 `${CMAKE_RUNTIME_OUTPUT_DIRECTORY}/${Config}`。故不需要对 `CMAKE_RUNTIME_OUTPUT_DIRECTORY` 变量画蛇添足，对其额外加上 `${CMAKE_BUILD_TYPE}` 等构建配置后缀。
7. **CMake** 设定了 `target_link_directories`，且链接库位置正确但仍然无法找到链接库的话，考虑直接使用链接库的绝对路径进行链接。（极少情况下会发生的问题）
8. **CMake** 使用 `target_link_libraries` 链接库时，不必要添加库的后缀名（如 `.lib`，`.a`），**CMake** 会根据生成环境自动选用合适的后缀名。
9. **CMake** 的子项目不必要都添加 `minimum_version_required`，只要在根项目中加入即可。即使添加了，也应该保证版本需求一致。
10. 使用 **MSVC** 编译器对包含`UTF-8`等非`ANSI`编码字符的代码编译报错时，可以通过添加 `/source_charset:utf-8` 和 `/execution-charset:utf-8` 编译选项解决编码问题。其中前者决定了编译器对于源码文件编码方式的“态度”，后者决定了字面量在二进制文件中实际的编码方式。或者使用 `/utf-8` 同时指定二者。
11. 如果链接库已正常配置，但使用 **MSVC** 编译器链接库时报错 `符号未定义`，需要着重注意程序或库的运行库链接方式（`MT/MD`）。在 **CMake** 中可以通过`set(CMAKE_MSVC_RUNTIME_LIBRARY "MultiThreaded$<$<CONFIG:Debug>:Debug>")`指定 `MSVC` 运行库使用 `MT` 链接方式。
12. 对于已有链接库，但其无 **CMake** 配置的话，推荐使用 `find_library(XXX_LIB NAMES XXX_LIB_NAME PATHS XXX_LIB_DIR)` 进行链接库的配置，而不是手动设置 `link_libraries_directory` 等等。
13. **OpenCV** 需要按照指定窗口方式显示图片时，需要先显式创建窗口再设置窗口属性，最后进行 `imshow` 才有效。
14. `scanf()`函数失败后不会消费失败的输入，在循环输入时，应在输入之前前尝试消费掉可能的失败输入。例如通过`while (getchar() != '\n);`。
15. `scanf()`函数返回成功赋值的个数。
16. 当程序发生错误，且复现随机成功时，首先考虑以下几种可能：`多线程问题`，`野指针问题`。

1. **CMake** 的 `copy` 与 `copy_if_different`指令对于目标路径的解释方式：当目标路径是一个**已存在的目录**时，其会将文件复制至此目录下，而当目标路径不存在时，其会将目标路径作为**文件路径**而非**目录路径**进行处理。故，如果需要将若干文件复制至一个不存在的目录下时，可以通过 `make_directory` 指令先进行目录创建。

## 2026-04-05

1. **CMake 生成器表达式** 如 `$<BOOL:VAR>` 等，其中变量 `VAR` 必须使用 `${VAR}` 进行展开解析。对于 `$<BOOL:VAR>` 而言，当其传入的值（字面量）为 `0`，`OFF`，`NO`，`FALSE`，`IGNORE`，空字符串，或以 `NOTFOUND` 结尾的字符串时返回 `0`，其余情况均返回 `1`。
2. **Qt** 的 GUI 操作必须在主线程中进行，不可以在子线程中进行。如果需要子线程对 GUI 进行更新。可以使用信号-槽系统在子线程中发出信号，在主线程的槽函数进行GUI更新（注意，此信号-槽连接需要使用队列连接）。或者使用队列连接的 `QMetaObject::invokeMethod`，其用于**跨线程或延迟调用一个 QObject 的槽函数或可调用方法**，而不需要显式建立信号-槽连接。本质上是通过 **Qt** 的元对象系统（Meta-Object System）动态调用方法。
3. **NDI SDK** 通过名称连接 NDI 源时，如果存在多个同名 NDI 源，则其连接的源是不确定的。
4. 在构建 `QListWidgetItem` 时，如果指定了 `QListWidget` 实例作为其父对象，那么其会被直接加入到父对象项中，而不需要再使用 `addItem` 进行添加。
5. 使用 `*deployqt` 对 **Qt** 程序进行部署时，如果未明确指定 `--release` 或 `--debug` 会尝试自动检测，但有时会检测出错（尤其是在多配置生成器中）。无论何时，都应该显式指定构建配置。
6. `QMessageBox` 通过 `QString` 创建自定义按钮时，无法使用 `QMessageBox` 相关的静态方法获取并判断返回值。此时，应该手动构建 `QMessageBox` 对象，并利用 `addButton()`，`clickedButton()` 判断按下的按钮。例如

    ```cpp
    QMessageBox msgBox;

    // ...

    QPushButton* btn1 = msgBox.addButton("Button 1", QMessageBox::AcceptRole);
    QPushButton* btn2 = msgBox.addButton("Button 2", QMessageBox::ActionRole);
    QPushButton* btn3 = msgBox.addButton("Button 3", QMessageBox::RejectRole);

    msgBox.exec();

    if (msgBox.clickedButton() == btn1)
    {
     // ...
    }
    else if (...)
    {
     // ...
    }
    ```

7. 不推荐在析构函数中调用成员函数，以防止未注意到的潜在问题，即使使用也应该仔细检查调用关系。析构函数应该明确其语义，手动对资源进行释放。
8. **OpenCV** 的 `imwrite()` 函数会覆盖已有文件。

## 2026-04-21

1. **线程安全**：多个线程同时调用不会把共享数据搞坏（常可用 `std::mutex` 实现）。

    **可重入**：函数在“尚未返回时再次进入”也安全（比如被中断/回调再次调用），不能依赖未保护的共享可变状态，也不能依赖会卡死的锁模式。

    **例子**：一个函数可以线程安全但不可重入：例如函数内部用普通 `std::mutex` 保护全局状态；并发线程调用安全，但若同一线程在持锁期间被再次调用，可能死锁，因此不可重入。

    **结论**：可重入通常是线程安全的，但线程安全不一定可重入。

6. 不要在基类的析构函数中调用纯虚函数的成员函数（包括链式调用），因为此时，派生类部分已经被销毁，此时纯虚函数的实现已不存在，将会抛出异常。更需要注意的是，在多线程程序中，即使从调用链路上没有纯虚函数的参与，也需要警惕以下情况：[多线程程序中基类析构与纯虚函数的潜在坑点]([C++]%20多线程程序中基类析构与纯虚函数的潜在坑点.cpp)。虽然基类的析构函数没有调用任何纯虚函数。但是，在基类的析构函数中调用 `stop` 函数并等待线程退出时，派生类已经被销毁。考虑这种情况：工作线程还在运行（将要被退出但还未退出），并且工作线程正在执行一个纯虚函数，而由于派生类的纯虚函数定义已经被销毁，那么此时程序将会抛出异常。解决办法是，在派生类的析构函数中调用 `stop` 函数。
7. 警惕在含有纯虚函数基类的构造函数中启动线程。因为如果工作线程中调用了纯虚函数的话，则有可能在派生类还未构造完成时调用纯虚函数，而这是未定义行为。
8. 错误码 `EINTR` 表示系统调用被信号中断，常见于 `read/write/poll/select/nanosleep` 等阻塞调用期间收到信号。通常重试该系统调用即可。
    错误码 `EAGAIN` 和 `EWOULDBLOCK` 表示 *现在做不了，稍后再试*，常见于非阻塞 `read/write/accept/connect` 和资源暂时不可用（例如当前没有数据可读）。通常可以使用短暂重试进行处理。对于非阻塞模型也可以返回上层，等 `poll/epoll/select` 之后唤醒。

8. **C/C++** `_Alignas` 和 `alignas`

    `alignas` 是 **C++11** 引入的对齐说明符，用来指定变量、数组、结构体/类、联合体等的对齐方式。

    **C11** 标准引入了 `_Alignas` 关键字，用法和功能与 **C++** `alignas` 相同。（包含头文件 `stdalign.h` 后也可以使用 `alignas` 宏，实际上就是 `_Alignas`）

    **MSVC** 编译器也有特定的扩展 `__declspec(align())`。

    **作用**

    - 提高缓存效率
    - 满足特定硬件要求（如 SIMD 指令需要对齐）
    - 优化内存访问性能

    **常见写法有两种**

    1. 按某个类型的对齐要求对齐

        ```cpp
        alignas(MyType) char buffer[128];
        ```

        意思是：buffer 的地址至少满足 MyType 的对齐要求。

    2. 按指定字节数对齐

        ```cpp
        alignas(16) char buffer[128];
        ```

        意思是：buffer 按 16 字节对齐。

10. **C/C++** 中的 **线程局部存储关键字** `thread_local`（仅 **C++11** 及以上），`__thread`（**GCC 5** 以下旧版本，**C/C++**），`__declspec(thread)`（**MSVC**，**C/C++**），`_Thread_local`（仅 **C11** 及以上）作用。

    用这些关键字修饰的变量，每个线程都拥有独立的副本，互不影响。

    **宏定义示例**

    ```cpp
    #if defined(__cplusplus) && __cplusplus >= 201103L
        #define MY_THREAD_LOCAL thread_local        // C++11 及以上
    #elif defined(__GNUC__) && __GNUC__ < 5
        #define MY_THREAD_LOCAL __thread            // GCC 5以下旧版本
    #elif defined(_MSC_VER)
        #define MY_THREAD_LOCAL __declspec(thread)  // MSVC (Visual Studio)
    #elif __STDC_VERSION__ >= 201112L
        #define MY_THREAD_LOCAL _Thread_local       // C11 标准
    #else
        // ...
    #endif
    ```

    **使用示例**

    ```cpp
    // C++11
    thread_local int error_code = 0;
    thread_local std::string thread_name;  // 支持复杂类型

    // C11
    _Thread_local int error_code = 0;

    // GCC
    __thread int error_code = 0;

    // MSVC
    __declspec(thread) int error_code = 0;
    ```

11. **C/C++** 将一个表达式强转为 `void` 可以声明此表达式是有意不被使用的（或者理解为我已经用过了，但是不关心它的值），以防止编译器警告。可以通过以下宏定义进行多环境支持。

    ```cpp
    #ifdef _MSC_VER
        #define MY_NOTUSED(v) (void)(v)
    #else
        #define MY_NOTUSED(v) (void) sizeof(v)
    #endif
    ```

    对于非 **MSVC** 编译器，通过使用 `sizeof` 可以避免产生副作用（因为 `sizeof` 只看表达式类型而不会对表达式求值）（这一操作更像是在语法层面提及一下这个表达式）。

    而对于 **MSVC** 编译器，它并不总是将 `sizeof(v)` 视为“已使用”，导致有时仍然可能会报 `unused warning`，所以去掉 `sizeof` 的使用以保证警告能被稳定压制。

13. **Git** 在遇到提交冲突时，变基拉取和合并拉取的异同与适用途径。

    **变基拉取**

    `git pull --rebase`

    将本地提交"嫁接"到远程最新提交之上，历史记录是一条直线。

    ```plaintext
    远程: A - B - C
    本地: A - B - D（你的提交）

    变基后本地: A - B - C - D
    ```

    **合并拉取**

    `git pull`（默认合并拉取）

    将远程和本地分支合并，产生一个合并提交，历史记录有分叉。

    ```plaintext
    远程: A - B - C
    本地: A - B - D（你的提交）

    合并后本地: A - B - C - D - M（M是合并提交）
                     \     /
                      C - D
    ```

    **区别对比**

    | 对比项 | 变基拉取 | 合并拉取 |
    | ----- | ------- | ------- |
    | 历史记录 | 线性，干净 | 有分叉，保留完整历史 |
    | 合并提交 | 无 | 有 |
    | 提交的 SHA | 会改变（上面示意图的 D，本地分支的父提交是 B，变基拉取后父提交变为 C，SHA 自然会变） | 不变（远程分支的 C 和本地分支的 D 父提交不会发生改变，只是新增了一个合并提交 M） |
    | 适用场景 | 个人分支、保持整洁 | 团队协作、保留真实历史 |
    | 风险 | 不要在共享分支上用 | 较安全 |

    **建议**

    - 个人开发分支 → 用 `git pull --rebase`，保持提交历史整洁。
    - 团队共享分支 → 用默认 `git pull`，避免重写他人历史。

3. **C++** `delete` 只释放了指针所指向的内存，指针变量本身的值（那个地址数字）确实还在，但 **C++ 标准** 规定：访问或使用一个已释放对象的指针值（除了对其赋新值）即是 **UB**。

    **原因有如下几层**

    1. 标准层面

        **C++ 标准** 明确规定，对象的生命周期结束后，持有其地址的指针变成了 **invalid pointer value**，对其进行任何操作（包括比较 `==`）均为未定义行为，唯一的例外是对该指针变量本身赋值。

    2. 硬件/运行时层面

        - 某些平台（如带 **MTE** 的 **ARM**）会给内存打标签，`delete` 后标签失效，读取该指针会触发硬件异常。
        - **AddressSanitizer** 等工具会在 `delete` 后立即将该地址标记为 **已中毒**，后续对该指针的比较操作会被检测到并报错。

    3. 编译器优化层面

        编译器可以假设程序不存在 UB，并据此做优化：

        ```cpp
        delete pointer;
        if (other == pointer)   // 编译器可能认为这个比较"永远为 false"
        ...                     // 然后直接删掉这段代码！
        ```

        因为在合法的程序中，`other` 不可能等于一个已失效的指针，编译器有权利用这个假设进行激进的死代码消除。

4. **Qt** `FileDialog` 的 **filter** 文本格式如下：

    "Images (*.png *.jpg *.bmp);;All Files (*)"

5. **C++** 在进行模板编程时。假设有如下实现：

    ```cpp
    template <typename T>
    class A
    {
    public:
        // Mistake
        A(T&& value) : value_(std::forward<T>(value)) {}

        // Correct 1
        // A(T value) : value_(std::move(value)) {}

        // Correct 2
        // template <typename U>
        // A(U&& value) : value_(std::forward<U>(value)) {}

    private:
        T value_;
    }
    ```

    在上面的实现中，错误写法的错误原因在于它使用了类的模板参数 `T` 而非函数模板参数。因为 `T` 不是在函数调用点被推导出来的，所以错误写法的结果就是其参数类型 `T&&` 是右值引用类型而非万能引用类型，这导致其只接受右值实参而无法接受左值实参。

    对于后面种正确写法而言。方法一是直接按值传入，而方法二则是通过将构造函数自身作为函数模板实现了完美转发（万能引用）。

    两种正确写法的区别：

    | 场景 | 方法一 `T value + move` 按值传入 | 方法二 `U&&` 万能引用实现完美转发 |
    | --- | ------------------------------- | ------------------------------- |
    | 传入左值 `lvalue` | 1次拷贝 + 1次移动 | 1次拷贝 |
    | 传入右值 `rvalue` | 1次移动 + 1次移动 | 1次移动 |
    | 传入字面量（如 "hello"） | 1次构造 + 1次移动 | 1次构造（原地构造） |
